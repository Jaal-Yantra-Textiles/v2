# Agent Daemon — Best Practices (append-only)

- When the queued next task is the "big risky" slice (live-path rewrite, prod Dockerfile, ambiguous), prefer a clean adjacent low-risk slice the same issue already lists — e.g. deleting confirmed-dead code (0 refs repo-wide) is a legit, mergeable chunk and removes a "edit the wrong file" hazard. Verify deadness with BOTH the filename grep AND a grep for each exported symbol across `src` (exclude `.medusa/server/**` = build output) before deleting; also fix any stale comments that name the removed file.
- If the daemon's own uncommitted files (e.g. `scripts/agent-daemon/launch.sh`) block `git checkout -b … origin/main`, do NOT stash/reset (forbidden — nukes the SDK). Instead `git add <that one daemon file> && git commit` it onto the current daemon branch to preserve it, then branch off origin/main with a clean tree. A modified submodule gitlink (`apps/storefront-starter`) does not block the checkout and carries over harmlessly — never stage it.
- `scripts/` is UNTRACKED on `origin/main` (the 215MB daemon SDK lives there). After branching off main it shows as `?? scripts/`. Stage feature files by explicit path only; `git rm` already stages a deletion so don't re-`git add` the deleted path (pathspec won't match the gone file).
- zod v4 (this repo, 4.3.6) `z.record(z.any())` is malformed — leaves the value schema undefined → validateBody throws `Cannot read properties of undefined (reading '_zod')`, but ONLY when the field is non-empty (an empty object short-circuits validation, so a partial integration test can pass while the real path 500s). Always use the two-arg form `z.record(z.string(), z.any())`. Capture the real 500 body via a try/catch + console.log in the spec — the custom errorHandler suppresses console logging under TEST_TYPE, so a failing assertion alone won't show the stack.
- Stacked PRs for dependent backend slices: when a follow-up needs files from an unmerged PR (e.g. a registry that only exists on the parent branch), branch off that parent branch and `gh pr create --base <parent-branch>`. GitHub auto-retargets the stacked PR to main once the parent merges — keeps the diff focused on just the new slice. (#471 on #470.)
- Zod `.partial()` does NOT strip a field's `.default()`: `panelBaseSchema.partial()` where a field is `z.record(...).default({})` still yields `{}` for that key when the client omits it. A route guard like `if (data.x || data.y)` then fires even on updates that never touched x/y — e.g. the admin stats-panel PUT re-validates `operation_options` on every metadata-only update and 400s when the panel's options are required. When extending such a route, mirror the real client flow in tests (send the full payload) rather than fighting the latent default-injection; note it as a separate cleanup instead of widening the slice.
- An unmerged review backlog is a real signal: when the active-issue stack already has N PRs awaiting human review ("drain before adding more"), pick a *fresh independent* roadmap issue off main for the next chunk instead of stacking a 3rd dependent PR. A self-contained backend restore (delete-then-readd from git history) verifiable with a per-file spec is the ideal low-risk fill. Pull deleted code back verbatim with `git show <commit>~1:<path>` and re-check WHY it was removed (here #284 deferred to the very issue being built; #283 had dropped a *different* gate on purpose — don't reinstate that one).
- Zod `.partial()` does NOT remove a field's `.default()` — defaults still fire on omitted keys, silently injecting values into "partial" update payloads (real data-loss + 400 source). Any create-base schema reused via `.partial()` for updates must re-declare its defaulted fields as `.optional()` on the update variant. Verify the interaction empirically with the project's own `@medusajs/framework/zod` (a throwaway .mjs under apps/backend) before assuming behavior. (PR #473)

- When fixing a Zod `.partial()`+`.default()` injection bug, grep the whole `src/api` for the *same* pattern (`grep -rl ".partial()"` ∩ `.default(`) — it is rarely a single instance. Verify empirically (`base.partial().parse({})` in a throwaway node script) that the default actually injects, and confirm the route spreads `...validatedBody` before claiming a clobber bug; transient action-triggers (default=safe no-op) and nested per-item defaults are NOT bugs.
- When the review backlog is full of stacked PRs, pick an independent off-main roadmap item whose acceptance is fully specified in the issue (e.g. #377's domain-precedence order) and whose core logic is a PURE function — unit-testable with a stubbed query.graph, no DB, no migration, no Playwright. Ships clean without adding integration-test/dev-server risk.
- When the daemon review backlog is full (many unmerged stacked PRs) and all remaining queue items are blocked/gated/stacking, an ANALYSIS chunk that writes a *build-ready* design doc (exact model/route/test/wiring + carried gotchas) for the next queued slice is higher-leverage than stacking a 7th PR — it lets the next session build without re-discovery. (chunk 8/8, #457 audit-log)
- New-route import depth bites: an API route under `[id]/run/` is one dir deeper than a sibling list route, so `../` counts differ by one (6 vs 5 to reach `src/modules`). An off-by-one shows up as a single TS2307 over baseline — diff the count against a working sibling import, don't eyeball it.
- `medusa db:generate <module>` emits a `.snapshot-<module>.json` next to the migration — stage it too (every existing module tracks its snapshot); forgetting it leaves the next generate dirty.
- To integration-test a new audit/side-effect on an existing route, extend that route's existing spec file rather than adding a new one — reuses the shared-DB boot and dodges the per-file TRUNCATE-vs-CONCURRENTLY deadlock. A guard-path call that returns 200 with an inner error (e.g. bulk recalc vs a missing id) is a zero-seed way to exercise the write.
- A registry-of-jobs pattern (e.g. #457 maintenance-jobs) is the ideal daemon slice when the review backlog has an unmerged PR touching the SAME feature: adding an Nth entry to the registry array + its pure helpers conflicts ONLY if the prior PR edited that exact file. Check `git diff origin/main...origin/<unmerged-branch> --name-only` first — if registry.ts isn't in it, you can branch off main, add the job, and stay conflict-free (no stacking, no rebase risk).
- Ops-registry unit specs live under `src/api/.../__tests__/*.unit.spec.ts` — run with `TEST_TYPE=unit` (the `integration:modules` testMatch only globs `src/modules/*/__tests__/`). A pure-helper accumulator can have a different internal field name (`.date`) than the function return shape (`order_date`); annotate them separately or tsc throws TS2339.
- When branching off `origin/main` to extend a registry that sibling unmerged PRs also touch (e.g. #457 MAINTENANCE_JOBS), expect a NEW utils import (`Modules`) to be missing on main even though it exists on the sibling branch — `tsc` catches it as TS2552; add the import. Off-main = registry has only the merged jobs, so mirror the main version's test file (no sibling-branch test refs like `res.data.audit`).
- #484 partner search/filter: the partner-ui search box is fine (URL `q` → SDK `query`); the bug is backend `/partners/*` list routes that never read `req.query` and return the full list. Always check the backend route reads `q` before assuming a UI bug. The reusable fix pattern is an in-app post-filter on sku/title (see partners/inventory-items/raw-materials route).

- When draining the #484 partner-search route fixes, branch each follow-up off `origin/main` (NOT off the prior chunk-branch) so PRs stay independent for human review; the in-app `q` post-filter pattern (read q/limit/offset → filter → slice → count=matched) is copy-paste from PR #487 inventory-items for any route that already loads its full set in memory (customers, raw-materials).
- When a partner list route paginates inside `query.graph` but filters status/q in-app, the filter runs on a single page and `count` is per-page (page-vs-set bug). Fix = drop `query.graph` pagination, fetch the full partner-scoped set, then filter+paginate in a pure helper (count = total matched). Extracting that helper makes it unit-testable (`*.unit.spec.ts`, `TEST_TYPE=unit`) — far cheaper than an integration test that needs a fully linked inventory order. `req.validatedQuery` is typed `unknown` on partner routes whose middleware borrows the admin schema; cast `as <LocalQueryType>` when destructuring or tsc TS2322 fires on any typed consumer.
- #484 designs route: the page-vs-set bug recurs anywhere a /partners/* list paginates inside query.graph (skip:offset/take:limit) before an in-app merge/post-filter — fix is always: fetch full set (capped take), filter+paginate in a pure unit-tested helper, count=total matched. 4th identical fix (#487/#488/#489/#490).

- **#484 partner-search has 5 backend list surfaces, all with the same root cause** (route never reads `q`; some also paginate before filtering = page-vs-set per-page count). Fix pattern that worked across #487–#491: read `q`, fetch the full partner-scoped set, then filter+paginate in a PURE helper (`apply<X>ListFilters`) with count=total matched, and a `TEST_TYPE=unit` spec on the helper — avoids booting Medusa for heavily-linked fixtures (reservations/orders need inventory-item+stock-location+sales-channel links).
- Multi-store currency trap (#485): in a now-multi-tenant deployment, `query.graph({entity:"store"})` returns ALL stores and `stores?.[0]` grabs an arbitrary one (prod: the platform "JYT Medu Store" = EUR), not the partner's store (INR). Verify store cardinality with `GET /admin/stores?fields=*supported_currencies` before assuming "the store" is singular. Any `stores[0]` in a currency/region path is a latent bug.
- When a prod "native addon won't load" issue blames `--ignore-scripts`, first check the addon's `engines.node` + shipped `prebuilds/` ABIs vs the Docker base image's Node ABI. Modern addons (isolated-vm v6 = prebuildify) need NO build tools — they need a matching prebuild. A Docker `docker run --rm node:<ver>-slim` one-liner that `npm i`s the pkg and requires it proves load/fail in seconds without a full image build. (#494: isolated-vm@6.1.2 needs Node>=22, prod was node:20-slim.)
- A many-to-many *within* a single Medusa module (e.g. customer↔customer_group) is NOT a `remoteLink` target — `remoteLink.create/dismiss` 500s on the undefined link. Use the module service's own methods (addCustomerToGroup/removeCustomerFromGroup) or the core workflow that wraps them (linkCustomerGroupsToCustomerWorkflow). Read the admin route first to confirm (#495/PR #500).

- **Single shared workspace + concurrent agents = branch-switch collisions.** If another process checks out a different branch between your `git checkout -b` and your `git commit && git push`, your commit lands on the WRONG branch and the push pushes the stale HEAD → `gh pr create` fails "No commits between main and <branch>" and your working-tree edits look reverted. They're NOT lost: `git reflog` shows your commit hash; `git checkout <your-branch>` (still at origin/main) then `git cherry-pick <hash>` recovers your exact delta. Mitigation: commit+push in ONE chained command right after editing, and always verify `git log --oneline origin/main..HEAD` shows your commit before `gh pr create`.
- When moving a dep between `dependencies`/`optionalDependencies` in package.json, the Dockerfile builds with `pnpm install --frozen-lockfile` — so `pnpm-lock.yaml` MUST be updated too or the prod build fails. Hand-edit the lockfile (no repo-root `pnpm install`): move the entry between the importer's `dependencies:`/`optionalDependencies:` blocks (same spec/version); the package snapshot stays unchanged. Validate with `node -e "require('js-yaml').load(...)"`.
- #485/#457: when a backfill must re-stamp an order column, confirm the column is in `UpdateOrderDTO` (e.g. `currency_code?`) AND that `orderService.updateOrders([{id,...}])` is already used in-tree before assuming the mutation is allowed — order is heavily guarded. Resolve order→partner via the link table `partnerOrderLink.entryPoint` (filter `partner_id`), NOT the `partner.orders` graph accessor (pluralisation not guaranteed; per list-partner-orders.ts).

- **Before building a "rewire all N call sites" forward-fix, audit each site for whether the needed context is actually in scope.** #485 was framed as "rewire 8 `stores[0]` sites"; the audit found only 3 were genuine duplicated currency scans, 1 was already correct, and the rest were storefront (not currency) — and partner is never in scope at currency-stamping time anyway (linked post-creation). The honest deliverable was a behaviour-preserving centralisation + a memory note correcting the framing, not a risky partner-plumb that cannot work.
- UI-slice verification: if `yarn dev` is already running (check `lsof -iTCP:9000`/`:5173`) you CAN Playwright-verify admin UI — but the admin needs a login. No creds are in `.env`; create a throwaway admin with `npx medusa user -e x@jyt.local -p Pass123!` (additive to the local dev DB, operator deletes later). Admin login fields are `input[name="email"]`/`input[name="password"]`, served at `localhost:9000/app`, custom settings routes at `/app/settings/<folder>`.
- Admin custom UI that consumes an already-merged API does NOT need to stack on unmerged backend PRs — branch off `origin/main` for a clean independent PR (the UI is generic over the backend contract).
- Design-first chunks: write the notes doc + post the open decisions as an issue comment (✅-marked recommendations the user can veto), then SHIP THE DOC AS A DOCS-ONLY PR off origin/main and STOP. Don't build past the decisions. Mirrors the #485/#494 steer loop; keeps the daemon unblocked without guessing on a data-model/UX fork.
- `medusa db:generate <module>` runs WITHOUT a live DB (diffs models vs the `.snapshot-*.json`) and emits idempotent `add column if not exists` ALTERs itself — handy to regenerate the snapshot, but per the create-if-not-exists hazard keep your hand-written migration and just delete the generated dup, retaining the regenerated snapshot.
- Stacked slices: when slice N depends on an unmerged slice N-1, branch off N-1's branch and open the PR with `--base <slice-N-1-branch>` so the PR diff is only slice N (not N-1's already-reviewed code).
- cwd-shift footgun: a `cd <dir> && …` Bash call leaves the persistent shell cwd in `<dir>`; a subsequent Write can land a file one level too deep (e.g. `apps/backend/apps/backend/…`). Always `cd` to an absolute path at the top of each Bash block and `git status` before staging. The HTTP integration runner takes a NAME substring, not a path: `pnpm test:integration:http:shared -- ops-maintenance-batches` (a full path → "0 matches").
- A Medusa route file can export multiple HTTP methods: add `GET` to an existing `batches/route.ts` next to `POST` for the index/list (no new file), and only spin up `batches/[id]/route.ts` for the param detail. Each method needs its OWN middleware matcher entry (same path, different `method`). #508 slice 3.
- #508 jobs (slice 4+) are registry-only appends to `registry.ts` + `MAINTENANCE_JOBS` — they do NOT depend on the batch stack (slices 1-3 never touch `registry.ts`), so branch each off `origin/main` as an INDEPENDENT PR (no stacking). Verify with `git diff origin/main..<slice-branch> --stat -- .../registry.ts` (empty = safe to branch off main).
- When a "backfill" job targets a denormalized COLUMN that mirrors a module LINK, scan FROM the link table (`link.entryPoint`, query.graph fields = the remoteLink.create payload keys, e.g. `production_runs_id`/`consumption_log_id`) — it's far smaller than all null-column rows and is exactly the drift population. Verify which side is the actual READ path first (cost-summary read the column, not the link), so you know which to backfill into. Fill only when the link is unambiguous (exactly one); never overwrite an existing value (idempotent); surface multi-link "ambiguous" as a summary skip, not an errors[] entry (mirror the prior job's skip-clause convention).
- A Playwright-gated UI slice can still yield a clean BUILD chunk headless: split out the pure data layer (react-query hooks render nothing → no screenshot needed, tsc-verifiable) as its own PR + ship an implementation-ready render spec, instead of analysis-only. De-risks the next UI session while keeping the daemon productive.

- When the active issue's only remaining slice is Playwright-gated UI (headless daemon can't verify), look upstream for a backend gap the UI will need (e.g. a missing query filter the render spec implies) — that's a clean, integration-testable slice that unblocks the UI without needing dev/Playwright. (#508 → PR #517: added `batch_id` filter to GET /runs because the "All runs" tab needs single-job-only rows.)
- To start an independent PR off origin/main while sitting on an unmerged feature branch that edited a shared file (e.g. ops-maintenance.ts also touched by an open PR): `git checkout -- <shared-file>` to drop your working edits, THEN `git checkout -b new origin/main` — git cleanly resets the now-unmodified shared file to main's version, while keeping your edits to files whose HEAD==main. Avoids dragging the other PR's content in. Defer the shared-file edit to the PR that already owns it.

- Medusa validated routes (`validateAndTransformQuery` + zod) REJECT unknown query params with a 400 — they are NOT silently ignored. So an admin UI must not send a param the *live* backend route doesn't accept yet: a forward-compat field added for an unmerged backend PR (#508 render slice's `batch_id`, awaiting #517) will 400 the call and break the tab. Decouple: don't send it until that PR merges; leave a one-line TODO. (Caught only because dev was up + Playwright surfaced the 400.)
- A Playwright-gated UI slice IS buildable headless when the operator's `yarn dev` is already up (check `lsof -iTCP:9000`/`:5173`). Two setup gotchas: (1) the local dev DB often lacks merged-to-main migrations → `TableNotFoundException` on new-module tables; `npx medusa db:migrate` syncs it (additive). (2) Seed history with a NO-OP applied batch (jobs with huge thresholds → 0 changes, e.g. `prune-ops-audit-runs older_than_days:9999`) via `POST .../batches {dry_run:false}` — exercises the full UI without mutating real local data.
- Robust admin login in Playwright: `waitForURL('**/app**')` matches `/app/login` immediately and returns before auth completes. Instead `await Promise.all([page.waitForResponse(r=>r.url().includes('/auth/')&&r.request().method()==='POST'), page.click('submit')])` then `waitForFunction(()=>!location.pathname.endsWith('/login'))`. Password field selector `input[type="password"]` is more robust than `input[name="password"]`.
- Medusa admin vs partner-ui ports: the **admin** (extension routes like /stats, Settings consoles) is served by the backend at `http://localhost:9000/app`. Port **:5173 (Vite) is the separate partner dashboard** ("Welcome to Partner Dashboard") — its router 404s on `/app/*`. Always Playwright-verify admin UI against `:9000/app`, not `:5173`.
- When this Mac's python3.14 is broken (pyexpat dlopen on import), skip the Python webapp-testing path: `npm install playwright` in **/tmp** (never repo-root — sweeps the 215MB SDK) and drive with Node ESM; chromium is already cached in `~/Library/Caches/ms-playwright`.
- Target a specific Medusa `ActionMenu` (the "…" button) by scoping to a sibling: e.g. a stats panel card's edit menu is `div:has(> button[aria-label='Refresh']) button[aria-haspopup='menu']`. A bare `button[aria-haspopup='menu'].first()` matches the top-left store/user menu instead.
- Partner routes have NO global auth: `/partners*` middleware only adds CORS + locale; every new partner route needs its OWN `authenticate("partner",["session","bearer"])` matcher in `src/api/middlewares.ts` (a GET with no body needs nothing else). Forgetting it = route runs unauthenticated.

- #337 partner parity slices follow a tight recipe: copy the admin route's data query verbatim (`listDesignComponents`, lineage walk, etc.), wrap with `assertPartnerOwnsDesign`, then add a PURE `scope*ToPartner` helper (filters by `owner_partner_id`) with its own unit spec under `partners/designs/__tests__/`. Self-referential design_component: `used-in` = query by `component_design_id` + relation `parent_design`; scope on `parent_design.owner_partner_id`.

- #337 `components` (PR #527) completes the design_component pair: `components` = query by `parent_design_id` + relation `component_design`, scope on `component_design.owner_partner_id` (mirror of `used-in`). When mirroring an admin route that has GET+POST, copy ONLY the GET for partner read parity — partner CUD on shared BOM structure is a separate (often locked) decision.
- #337 parity: confirm whether prior series PRs MERGED before assuming you must stack — `git show origin/main:<central-file> | grep -c <prior-matcher>`. Once the series base merged, later slices branch independently off origin/main (no stacking, no conflict). The #337 revisions/used-in/components matchers were all on main, so #528 (tasks) branched clean.
- A handoff-queued "next read-parity slice" can be wrong: verify the admin route's HTTP method before assuming it's a GET mirror. `admin/designs/:id/segment` was queued as a "read" but is a POST fal.ai op. Cheap check: `grep -oE "export (const|async function) (GET|POST|PUT|DELETE)" <route>` per admin subroute — discovers method + reveals whether the clean-read series is actually exhausted.
- Partner AI routes are quota-gated, NOT naive admin mirrors. Before mirroring an admin AI/fal endpoint, check `modules/ai_usage` — add a `MONTHLY_QUOTA` key (no migration, `operation` is a free string) and copy describe-image's checkQuota→402→recordUsage(before-call)→fal pattern. This keeps a "decision-bearing" AI route buildable without a new product policy (the soft-paywall precedent IS the policy).
- Medusa middleware matchers are EXACT-path, not prefix globs: a matcher for `/x/:id/segment` does NOT auth-cover `/x/:id/segment/depth` — every nested subroute needs its own matcher block. (chunk 3, PR #530 depth route.)
- A batch of routes flagged "decision-bearing" by a prior handoff can still hide ONE clean buildable slice — don't accept the label without checking. Chunk 4 found `revise` (lumped with approve/notify-customer as "changes canonical design state") was actually safe: reading `reviseDesignWorkflow` showed it RE-LINKS the partner to the new revision (`linkPartnersToNewDesignStep`), so an own-design fork stays partner-owned with no external side-effects. The 30s of reading the workflow turned a "needs a decision" into a shipped PR (#531). Verify the actual side-effects (product creation? customer email? cross-tenant link?) before deferring a mutation as decision-bearing.
- When a chunk is mostly analysis but yields one clean build, do BOTH: ship the clean slice as a PR AND write the decision doc for the rest (`apps/docs/notes/337_PARTNER_DESIGN_MUTATION_PARITY.md`) so the deferred routes are teed-up (per-route decision + recommendation + build order), not just "decision needed". Next chunk/user resumes without re-deriving.
- Re-confirm a flagged "decision-bearing" route against its ACTUAL schema, not the JSDoc. Chunk 5 found `POST tasks` (deferred for an "assignee" cross-tenant worry) had NO assignee field in `AdminPostDesignTasksReq` — the JSDoc's `assigneeId` example was stale. Reading the validator + workflow (~1 min) turned the 4th "decision-bearing" mutation into a shipped clean own-design slice (PR #532), same as `revise` (#531). Two of the five "deferred" #337 mutations were actually clean.
- When a chunk concludes "no clean buildable slice" because a candidate looks risky/destructive, CHECK FOR AN EXISTING MERGED PRECEDENT first. `repair-inventory-raw-material-links` (PR #533) looked like a risky novel link-mutation build until I found the merged `repair-partner-region-links` (#514) doing the exact same orphan-enumerate-then-dismiss pattern — turning a "switch to analysis" call into a clean buildable mirror. A precedented destructive op is low-risk; an unprecedented one is the risky one.
- Before writing a data-plumbing backfill, check what `NULL` MEANS for the column: if NULL is a valid semantic state (e.g. `design.owner_partner_id IS NULL` == admin/canonical, used as a filter discriminator), backfilling it from a weaker signal (an assignment link) is HARMFUL re-scoping, not a fix — the "job" is a non-job. (chunk 7, #336 analysis)
- New Medusa module needs its resolve path in BOTH `medusa-config.ts` and `medusa-config.prod.ts` (the partner-payment-config precedent does NOT register in `.dev.ts`); a hand-written create-table migration needs no `.snapshot-*.json` (snapshots only feed `db:generate` diffing). For bigNumber money cols, the migration must add a `raw_<col>` jsonb sidecar alongside the `numeric` col, and enums become `text check (col in ('a','b'))` — mirror `internal_payments/migrations/Migration20250105072006.ts` rather than guessing.
- When a slice depends on code that only exists on an unmerged prior-slice branch (e.g. Slice 1 imports `compute-fee.ts` shipped in Slice 0's PR), you MUST stack on that branch (not origin/main) even if there's no shared-file conflict — otherwise the import target is missing and the slice can't build standalone. This is dependency-stacking, distinct from shared-registry-file-conflict stacking.

- **Resolve a partner FROM an order_id:** query the D3 link table directly by entryPoint — `query.graph({entity: partnerOrderLink.entryPoint, fields:["partner_id"], filters:{order_id}})` (import the default export of `src/links/partner-order.ts`). No row ⇒ retail order ⇒ skip. Don't trust a `partner.orders`/`order.partners` graph accessor (pluralisation not guaranteed). Same pattern reversed (`fields:["order_id"], filters:{partner_id}`) lists a partner's orders.
- **Test an `order.placed`/`order.canceled` subscriber without the event bus:** create the order via `container.resolve(Modules.ORDER).createOrders({currency_code,email,items:[...]})`, create any link via `container.resolve(ContainerRegistrationKeys.LINK).create([{[PARTNER_MODULE]:{partner_id},[Modules.ORDER]:{order_id},data:{...}}])`, then call the exported handler directly: `await handler({event:{data:{id}}, container} as any)`. Mirror `integration-tests/http/order-placed-production-runs.spec.ts`. Per-file run only (`pnpm test:integration:http:shared -- <spec>`).
- #336 reversal/compensation slices: put the idempotency+gating logic in a SERVICE method (`reverseFeeForOrder`) not the subscriber — keeps the subscriber a thin best-effort wrapper and the logic unit/integration-testable; return null when nothing changed so the subscriber logs only real transitions. For cancel, reverse ONLY `accrued` rows (skip invoiced/waived/reversed) so you never silently undo a billed fee.

- **Read-API roll-ups: compute over the full filtered set, slice for pagination AFTER (#484 page-vs-set).** For `GET .../[id]/fees` (#336 Slice 4) the summary is built from `listPartnerFees(filters)` then `all.slice(offset,offset+limit)` for the page — never summarize the page or counts go per-page. Pure summarizer (`summarizePartnerFees`) is unit-tested in isolation; the route just wires it. Partner-side mirror = same handler body + `getPartnerFromAuthContext` 403-on-mismatch guard.

- **Historical "catch-up" backfill jobs must mirror the live subscriber's net outcome, not just its accrue path.** #336 Slice 5 (`backfill-partner-order-fees`) skips `canceled` orders because the order.placed accrual subscriber (Slice 2) + order.canceled reversal (Slice 3) net to zero for them — accruing-and-leaving a canceled order `accrued` would over-count partner net commission. Encode that as a pure `shouldBackfillOrderFee(existingFee, orderStatus)` so the eligibility decision is unit-tested without the DB. When a registry job re-runs the same flow as a subscriber, reuse the SAME pure helpers (`resolvePartnerFeeRate`/`computeFee`/`findFeeForOrder`) so math/idempotency can't drift, and copy the closest existing job (`backfillPartnerOrderCurrencyJob`) wholesale for the partner↔order-link enumeration + dry-run/apply/errors[] envelope.
- #522: stats panel compute is `resolvePanel` (modules/stats/resolver.ts) dispatching to visual-flows `operationRegistry` (modules/visual_flows/operations/), NOT a stats-local executor — metric/series logic lives in aggregate-data.ts/time-series.ts. seed-stats-dashboards.ts SKIPS existing dashboards, so editing a panel def there never retro-patches live DB rows.
- UI-slice gotcha: a freshly-merged module's table may be ABSENT from the local dev DB → the route 500s with `TableNotFoundException` (not the UI's fault). Run `cd apps/backend && npx medusa db:migrate` (additive, safe while `yarn dev` runs) before Playwright-verifying, then seed sample rows via `npx medusa exec` resolving the module service. Before writing "order-test coverage", grep existing integration specs — #541's accrual/reversal/read-API were already fully covered; don't add redundant tests.

- **Migration class names must be globally unique across ALL modules.** Medusa tracks executed module migrations by class name in a tracking table shared across modules; two modules with the same `MigrationYYYYMMDDHHMMSS` → the 2nd (alphabetically-later module) is silently SKIPPED as "up-to-date" and its columns never land (incl. prod). Bit #348-A (partner) vs #604-A (designs), both `Migration20260622120000` → `partner.tax_id` never landed anywhere (PR #661 renamed partner→...120001). When hand-writing an ALTER migration on the same day as another, bump the timestamp seconds.
- **When an integration test fails on a "column does not exist" for a recently-merged column, isolate migration-vs-code FIRST:** `psql -c 'create database tmp'; DATABASE_URL=postgres://postgres@localhost:5432/tmp npx medusa db:migrate` then inspect `information_schema.columns`. If the column is missing after a REAL CLI migrate, it's a migration bug (e.g. name collision), not your code or the test harness. Avoids hours chasing the test-runner's DB reuse / stale `.medusa/server`.
